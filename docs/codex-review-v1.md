# Codex Security Review — git-haiku v1 (3502dd6)

1. **HIGH**, [tee.ts:36](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/tee.ts:36): `GITHAIKU_TEE=1` alone makes the app claim TEE mode for proof/key decisions even before proving the dstack socket works. Fix by separating config intent from verified TEE capability: fail startup if production/TEE mode is set and dstack key derivation plus quote/info calls do not succeed.

2. **HIGH**, [server.ts:64](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/server.ts:64), [server.ts:77](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/server.ts:77), [server.ts:55](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/server.ts:55): Non-haiku operational routes bypass the egress guard and return internal mode/provider/generator state, backend DID/policy, and raw attestation error messages. Fix by defining explicit public operational schemas, redacting error messages, and moving sensitive diagnostics behind owner/admin auth or startup logs only.

3. **HIGH**, [rate-limit.ts:20](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/rate-limit.ts:20): `/api/haiku` rate limiting is keyed by `ip|submittedCodeHash`, so an attacker can brute-force many different candidate codes from the same IP without hitting the per-code bucket. Fix with layered limits: per-IP global invalid attempts, per-owner/code limits after match, and preferably exponential backoff or temporary IP/code-family bans.

4. **MEDIUM**, [server.ts:231](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/server.ts:231), [audit.ts:44](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/audit.ts:44): Invalid-code audit records still hash every submitted guess into durable audit storage. That is not raw secret leakage, but it creates a durable oracle/enumeration trail; fix by rate-limiting before audit writes and coalescing invalid attempts by coarse IP/window instead of per-guessed code hash.

5. **MEDIUM**, [delegations.ts:87](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/delegations.ts:87): Delegation validation checks resource coverage, but does not verify the delegation audience/target is the backend DID before storing it. Fix by requiring the deserialized delegation target/audience DID to equal `getBackendIdentity().did`, and reject otherwise before persistence.

6. **MEDIUM**, [delegations.ts:99](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/delegations.ts:99): Expired delegations are accepted and stored; expiry is only returned as metadata. Fix by parsing expiry strictly and rejecting expired or unparseable expirations at `POST /api/delegations`.

7. **MEDIUM**, [frontend/src/lib/appManifest.ts:26](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/frontend/src/lib/appManifest.ts:26): The owner app manifest declares `GITHUB_TOKEN: ['read']` while the flow calls `tcw.secrets.put('GITHUB_TOKEN', token)`. Fix by granting the owner app the write/create capability required by `secrets.put`, and keep the backend delegated manifest read/decrypt-only.

8. **MEDIUM**, [server.ts:265](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/server.ts:265): The `/api/haiku` catch path returns `200` with `allowed:false` for operational failures after code validation. This preserves egress shape, but hides deployment failures and can make attested-service monitoring miss broken secret/RedPill/attestation paths; fix with guarded JSON plus appropriate non-2xx status for internal failures, if clients can tolerate it.

9. **LOW**, [server.ts:48](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/packages/backend/src/server.ts:48): CORS is `origin: true`, allowing any website to call owner endpoints and trigger nonce/signature prompts. SIWE protects the routes, but production should restrict allowed frontend origins to reduce phishing/CSRF-style prompting.

10. **LOW**, [Dockerfile:47](/Users/samgbafa/conductor/workspaces/tinycloud-dev/daegu/worktrees/git-haiku/githaiku-build-v1/Dockerfile:47): The runtime image copies the full build workspace and runs `tsx` from source. This is larger and carries more build-time surface than needed; fix by emitting JS during build, pruning dev dependencies, and copying only runtime artifacts.

**VERDICT:** Not safe to deploy as the attested git-haiku v1 yet. The egress guard for `/api/haiku` is directionally solid, SIWE nonce replay protection looks sound, and the main success response does not directly expose commit data. Before TEE deployment, fix the TEE-mode verification/startup failure behavior, redact or lock down non-haiku operational routes, close the rate-limit enumeration gap, and tighten delegation audience/expiry validation.
